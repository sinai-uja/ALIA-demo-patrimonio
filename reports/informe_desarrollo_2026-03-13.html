⚡ Pipeline RAG — Flujo Completo

1

Embedding de la Query

MrBERT genera vector 768d via POST /embed. Mean pooling sobre todos los tokens.

2a

Búsqueda Vectorial

pgvector <=> cosine distance. Top 20 chunks. Filtros: heritage_type, province.

2b

Búsqueda Full-Text

PostgreSQL tsvector + ts_rank_cd. Idioma: español. 37 stopwords. Top 20.

3

Fusión Híbrida — Reciprocal Rank Fusion (RRF)

RRF(d) = Σ 1/(k + rank + 1) con k=60. Texto ponderado ×1.5 (keywords exactas son señal fuerte en dominio patrimonial). Normaliza a [0, 1] invertido (0 = mejor).

4

Filtro de Relevancia

Umbral: score ≤ 0.35. Si 0 chunks pasan → ABSTENCIÓN (no se invoca al LLM, respuesta fija).

5

Re-ranking Heurístico (4 señales)

40% base 30% título 20% cobertura 10% posición — Output: top 5 chunks.

6

Ensamblado de Contexto

Máx 6,000 caracteres. Formato: [N] Título (tipo, provincia) + Contenido + Fuente: URL. Separador: ---.

7

Generación LLM

salamandra-7b-instruct via vLLM. System prompt: grounding estricto (7 reglas). Temp: 0.3. Max: 512 tokens. Auto-retry con tokens reducidos si error 400.

⚙️ Paso 1 — Embedding

// MrBERT genera vector 768-dimensional
query_embedding = [0.023, -0.118, ..., 0.045] // 768 floats

🔎 Paso 2a — Búsqueda vectorial (SQL)

SELECT id, document_id, title, heritage_type, province,
       municipality, url, content,
       embedding <=> :query_vec AS score
FROM document_chunks_v1
WHERE heritage_type = 'patrimonio_inmueble'
  AND province = 'Jaén'
ORDER BY score ASC
LIMIT 20

🔎 Paso 2b — Búsqueda full-text (SQL)

SELECT ...,
       ts_rank_cd(search_vector, plainto_tsquery('spanish', 'castillos provincia jaen')) AS score
FROM document_chunks_v1
WHERE search_vector @@ plainto_tsquery('spanish', 'castillos provincia jaen')
ORDER BY score DESC
LIMIT 20

🎲 Paso 3 — RRF Fusion

// Chunk que aparece #1 en ambas listas:
vector_rrf  = 1.0 / (60 + 0 + 1) = 0.0164
text_rrf    = 1.5 / (60 + 0 + 1) = 0.0246
total_rrf   = 0.0410  // máximo posible
normalized  = 0.0    // 1 - (0.041/0.041) = 0  (mejor score)

📈 Paso 5 — Re-ranking

// Chunk con "Castillo" en título:
base_score     = 1.0 - 0.0  = 1.0   // × 0.4 = 0.40
title_match    = 1.0          // "castillo" ∈ title → × 0.3 = 0.30
term_coverage  = 0.67         // 2/3 terms found → × 0.2 = 0.13
position       = 0.5          // neutral → × 0.1 = 0.05
final_score    = 0.88         // alto → será top resultado

📥 Pipeline de Ingestión

Flujo de ingestión Parquet → BD

1

ParquetDocumentLoader

Lee parquet con pandas. Extrae columnas: id, url, title, province, municipality, text. Resto → metadata dict (no persistido).

2

ChunkingService

512 tokens/chunk, 64 tokens overlap. Paragraph-aware (divide en \n\n). Word-level fallback para párrafos grandes. Evita chunks finales diminutos.

3

HttpEmbeddingAdapter

MrBERT via HTTP. Batch size: 8 chunks. Cada chunk se enriquece con metadata (título, tipo, provincia) antes de embeber.

4

DocumentRepository

INSERT INTO document_chunks_v1. Idempotente: chunk_exists() previene duplicados. Trigger PostgreSQL auto-genera search_vector (tsvector).

🧠 Heurísticas y Algoritmos

🎲 Reciprocal Rank Fusion (RRF)

RRF(d) = Σ wi / (k + ranki(d) + 1)

k = 60           // constante de suavizado
w_vector = 1.0   // peso búsqueda vectorial
w_text   = 1.5   // peso búsqueda textual (+50%)

Combina rankings sin calibrar scores. El peso extra en texto se justifica porque keywords exactas (ej. "castillo", "retablo") son señal fuerte en dominio patrimonial con terminología específica.

📈 Re-ranking Heurístico

score(chunk) =
  0.4 × (1 - cosine_dist)        // base
+ 0.3 × title_match(q, title)   // título
+ 0.2 × coverage(q, content)    // cobertura
+ 0.1 × 0.5                     // posición (neutral)

title_match: fracción de términos de la query que aparecen en el título del chunk. coverage: fracción de términos de la query presentes en el contenido. Tokenización con 23 stopwords españoles + verbos conversacionales.

🚫 Filtro de Relevancia

if chunk.score <= 0.35:  // pasa
if chunk.score >  0.35:  // descartado

if len(filtered) == 0:
    return ABSTENCION  // no invoca LLM

Patrón de abstención: si ningún chunk supera el umbral, el sistema responde con un mensaje fijo sin invocar al LLM. Esto previene alucinaciones en queries fuera de dominio.

✂️ Chunking Paragraph-Aware

chunk_size    = 512 tokens
chunk_overlap = 64  tokens

// 1. Divide por párrafos (\n\n)
// 2. Si párrafo > 512 tok → split por palabras
// 3. Solapamiento de 64 tok en fronteras
// 4. Evita chunks finales < umbral

Los chunks se enriquecen con metadata antes de embeber: se prepende título + tipo + provincia + municipio al contenido, mejorando la relevancia semántica del embedding.

🚨 Gaps y Próximos Pasos

Gaps críticos

ALTA G1 — Sin georreferenciación

No hay coordenadas lat/lon en los datos. Las rutas virtuales no pueden mostrarse en mapa. react-leaflet está en el frontend pero sin datos geográficos. Acción: geocoding por municipio+provincia, o scraping de fichas de la Guía Digital IAPH.

ALTA G2 — Tipología no filtrable

Campos ricos de tipología (characterisation, type, subject_topic, etc.) se descartan en la ingestión. No se pueden usar como filtros en la búsqueda vectorial ni textual. Acción: persistir columnas adicionales en document_chunks + indexar + exponer como filtros en API.

MEDIA G3 — Datos sucios

~270 registros tienen HTML/metadata mezclado en campos province y municipality. Esto puede provocar fallos en filtros de búsqueda. Acción: pipeline de limpieza pre-ingestión.

MEDIA G4 — Tests mínimos

Solo 20 tests unitarios (chunking + schemas). Sin tests de integración del pipeline RAG, sin tests de endpoints API, sin tests de frontend. Acción: tests de integración con fixtures de BD, tests e2e del pipeline.

MEDIA G5 — LLM sin fine-tuning

Salamandra-7b-instruct se usa en su versión base sin ajuste al dominio patrimonial IAPH. La calidad de las respuestas está limitada por el conocimiento genérico del modelo. Acción: fine-tuning con corpus IAPH (en progreso).
