4.1 Backend — instrumentación de los casos de uso

Commit 62a38d0 — 5 archivos, +235 / -10

AddStopUseCase y RemoveStopUseCase reciben opcionalmente un TraceRepository (inyectado vía composition/routes_composition.py) y, al terminar la operación, persisten una ExecutionTrace con:

  • execution_type = "route" y execution_id = route_id — clave para reconstruir el historial completo agrupando trazas
  • pipeline_mode = "route_add_stop" o "route_remove_stop" — nuevos modos
  • query describiendo la acción en lenguaje natural ("Add stop: <título>" o "Remove stop #3")
  • steps con cada etapa cronometrada: stop_addition_request, heritage_asset_lookup, narrative_generation (con system_prompt, user_prompt, raw_response, narrative_segment), route_update
  • summary con metadatos clave (paradas antes/después, posición, narrativa en chars, etc.)
  • feedback_value queda disponible para que el usuario pueda valorar la edición concreta
  • Identidad del autor: user_id, username, user_profile_type

RouteApplicationService.add_stop() y remove_stop() propagan username y user_profile_type desde el endpoint hacia los casos de uso, y el endpoint routes.py los lee del User autenticado.

4.2 Endpoint admin de historial por ruta

Commit 87be1ad — 7 archivos, +191

Nuevo endpoint GET /api/v1/admin/traces/by-route/{route_id} que devuelve la lista cronológica de todas las trazas asociadas a una ruta (la generación inicial más todos los add_stop / remove_stop), filtrando trazas de otros admins (cada admin solo ve las trazas que él mismo generó).

Estructura completa hexagonal

CapaCambio

APIGET /admin/traces/by-route/{route_id} con response model RouteHistoryResponse

Response shape

{
  "route_id": "...",
  "traces": [ { "id": "...", "pipeline_mode": "route_generation", ... }, ... ],
  "aggregate": {
    "total_events": 5,
    "generation_count": 1,
    "additions_count": 3,
    "removals_count": 1
  }
}

El recuento de adiciones, eliminaciones y generaciones se calcula en el endpoint inspeccionando pipeline_mode de cada traza, lo que permite a la UI mostrar métricas sin re-procesar la lista.

👍 Avance 2 — Feedback en el modal de edición (TODO J2)

Commit c21ba9d — 2 archivos, +63 / -17

El componente SearchStopModal (modal que se abre al pulsar "Añadir parada" en una ruta y permite buscar activos patrimoniales para insertarlos) ya muestra los pulgares thumbs up/down en cada resultado, replicando el comportamiento que ya existía en la página principal de búsqueda (cd589ed del 11 de abril).

Cambios clave

  • Importa FeedbackButtons, useFeedbackStore y useAuthStore
  • Pre-carga los valores existentes con loadFeedbackBatch("search_result", ids) al recibir resultados — los pulgares aparecen ya marcados si el usuario había votado antes
  • Cada tarjeta muestra los pulgares apilados verticalmente con la miniatura, en size="sm" para no romper la densidad visual
  • Tipo searchId añadido al estado local para gestionar la clave compuesta (search_id, document_id)
  • El layout de la tarjeta se reordena: miniatura + pulgares en columna izquierda, título y descripción a la derecha

Metadatos de feedback enriquecidos

Para distinguir el feedback de edición del feedback de búsqueda libre, los metadatos añaden:

CampoValorPropósito
context"route_edit"Clave para segmentar analíticas vs "search"
route_idid de la ruta editadaPermite cruzar con la traza correspondiente
search_idid de la búsquedaMantiene la clave compuesta de feedback existente
document_idid del activo patrimonialIdentificación del item votado
querytexto de búsquedaPara segmentación por intent
heritage_type / provincemetadatos del activoAnalítica por categoría
user_profile_typeperfil del usuarioAnalítica por perfil

El cambio cierra el TODO J2 directamente: la UJA podrá distinguir el feedback dado durante una búsqueda libre (context = "search") frente al dado durante una edición de ruta (context = "route_edit"), y por tanto medir si los resultados que el usuario añade efectivamente a una ruta tienen mejor o peor valoración percibida que los simplemente listados.
